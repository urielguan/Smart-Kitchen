## 2. 系统架构设计

### 2.1 整体架构概览

智慧厨房管理平台采用**微服务架构**，基于Spring Boot 3.2 + Spring Cloud 2023 + Spring Cloud Alibaba 2023技术栈，通过Nacos实现服务注册与配置管理，通过Spring Cloud Gateway实现统一API网关，采用MySQL 8.0共享数据库（表前缀隔离）+ Redis缓存的数据架构。

### 2.2 微服务拆分策略

#### 2.2.1 拆分原则

**领域驱动设计（DDD）原则**：
- 按业务领域边界拆分，每个服务对应一个业务域
- 服务内高内聚，服务间低耦合
- 每个服务拥有独立的数据域（通过表前缀隔离）

**共享数据库策略**：
- 采用单一MySQL数据库，通过表名前缀实现逻辑隔离
- 避免分布式事务复杂性，保证数据一致性
- 适配JVM内存限制（Xms256m-Xmx512m），减少连接池开销

**服务粒度原则**：
- 服务数量适中（9个业务服务），避免过度拆分
- 每个服务职责单一，边界清晰
- 支持独立部署、独立扩展

#### 2.2.3 服务依赖关系

**依赖关系说明**：

```
auth-service (认证授权)
    ↓ 被所有服务依赖（Token校验）

sys-service (系统基础)
    ↓ 被所有服务依赖（组织、员工、字典查询）

scm-service (供应链) → wms-service (仓储)
    采购订单收货 → 生成入库单

wms-service (仓储) → recipe-service (菜谱)
    库存查询 ← 菜谱计划审核

recipe-service (菜谱) → cook-service (烹饪)
    菜谱计划审核通过 → 生成烹饪任务

cook-service (烹饪) → sample-service (留样)
    烹饪完成 → 生成留样任务

cook-service (烹饪) → device-service (设备)
    烹饪启停 → 温度传感器启停

health-service (晨检) → sys-service (系统)
    晨检记录 → 员工信息查询
```

**服务间通信方式**：
- 所有服务间通信采用**HTTP RESTful**方式
- 通过Nacos服务发现，使用服务名调用（如：http://wms-service:8083/api/v1/wms/stock/query）
- 每个请求携带traceId，支持全链路追踪
- 避免循环依赖，保持单向依赖关系

### 2.4 服务间通信设计

#### 2.4.1 HTTP RESTful通信规范

所有服务间通信采用HTTP RESTful方式，通过Nacos服务发现实现服务名调用。

**调用示例（wms-service调用sys-service查询组织信息）**：

```java
// 通过服务名调用（Nacos服务发现）
@Service
public class WmsStockService {

    @Value("${service.sys.url:http://sys-service:8089}")
    private String sysServiceUrl;

    public OrgVO getOrgInfo(Long orgId) {
        String url = sysServiceUrl + "/api/v1/sys/orgs/" + orgId;
        // 携带traceId
        HttpHeaders headers = new HttpHeaders();
        headers.set("X-Trace-Id", TraceContext.getTraceId());
        headers.set("Authorization", "Bearer " + TokenContext.getToken());
        // HTTP调用
        ResponseEntity<Result<OrgVO>> response = restTemplate.exchange(
            url, HttpMethod.GET, new HttpEntity<>(headers),
            new ParameterizedTypeReference<Result<OrgVO>>() {}
        );
        return response.getBody().getData();
    }
}
```

**统一响应格式**：

```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": { },
  "traceId": "abc123-def456",
  "timestamp": 1710691200000
}
```

**错误响应格式**：

```json
{
  "code": "PARAM_ERROR",
  "message": "参数校验失败：物料名称不能为空",
```

### 2.6 安全架构设计

#### 2.6.1 认证授权流程（JWT）

**完整认证流程**：

```
1. 用户登录
   客户端 → POST /api/v1/auth/login {username, password}
   auth-service → 校验用户名密码（BCrypt）
   auth-service → 生成accessToken（2小时）+ refreshToken（7天）
   auth-service → 将Token存入Redis（key: token:{userId}:{tokenId}）
   响应 → {accessToken, refreshToken, expiresIn}

2. 接口鉴权
   客户端 → 请求Header携带 Authorization: Bearer {accessToken}
   Gateway → 拦截请求，校验Token（白名单路径跳过）
   Gateway → 解析JWT，验证签名和过期时间
   Gateway → 查询Redis确认Token有效性
   Gateway → 将用户信息注入请求Header（X-User-Id, X-Org-Id, X-Tenant-Id）
   业务服务 → 从Header读取用户信息，执行业务逻辑

3. Token刷新
   客户端 → POST /api/v1/auth/refresh {refreshToken}
   auth-service → 校验refreshToken有效性
   auth-service → 生成新的accessToken
   响应 → {accessToken, expiresIn}

4. 用户登出
   客户端 → POST /api/v1/auth/logout
   auth-service → 从Redis删除Token
   响应 → 登出成功
```

**JWT Payload设计**：

```json
{
  "sub": "1001",
  "username": "zhangsan",
  "realName": "张三",
  "orgId": 100,
  "tenantId": 1,
  "roles": ["kitchen_manager", "cook_supervisor"],
  "permissions": [
    "scm:supplier:view",
    "scm:supplier:create",
    "wms:stock:view"
  ],
  "iat": 1710691200,
  "exp": 1710698400
}
```

### 3.6 JVM参数与资源配置

#### 3.6.1 JVM参数说明

所有Java服务统一使用以下JVM参数：

```bash
JAVA_OPTS="-Xms256m -Xmx512m -XX:+UseG1GC -XX:MaxGCPauseMillis=200"
```

**参数说明**：

| 参数 | 说明 |
|------|------|
| -Xms256m | 初始堆内存256MB |
| -Xmx512m | 最大堆内存512MB |
| -XX:+UseG1GC | 使用G1垃圾回收器 |
| -XX:MaxGCPauseMillis=200 | GC最大停顿时间200ms |

#### 3.6.2 各服务资源分配建议

| 服务 | CPU | 内存 | 磁盘 | 说明 |
|------|-----|------|------|------|
| MySQL | 2核 | 2GB | 50GB | 数据库服务，内存越大越好 |
| Redis | 1核 | 512MB | 10GB | 缓存服务 |

```java
@RestController
@RequestMapping("/api/v1/wms/materials")
@Slf4j
public class MaterialController {

    @Autowired
    private MaterialService materialService;

    @GetMapping
    public Result<PageResult<MaterialVO>> list(@Valid MaterialQueryDTO query) {
        PageResult<MaterialVO> result = materialService.listMaterials(query);
        return Result.success(result);
    }

    @PostMapping
    @PreAuthorize("hasAuthority('wms:material:create')")
    public Result<Long> create(@Valid @RequestBody MaterialCreateDTO dto) {
        Long id = materialService.createMaterial(dto);
        return Result.success(id);
    }
}
```

**Service层规范**：

```java
@Service
@Slf4j
public class MaterialServiceImpl implements MaterialService {

    @Autowired
    private MaterialMapper materialMapper;

    @Override
    @Transactional(rollbackFor = Exception.class)
    public Long createMaterial(MaterialCreateDTO dto) {
        // 参数校验
        validateMaterial(dto);
        
        // 构建实体
        Material material = MaterialConverter.INSTANCE.toEntity(dto);
        material.setTenantId(TenantContext.getTenantId());
        material.setOrgId(UserContext.getOrgId());
        
        // 保存
        materialMapper.insert(material);
        
        log.info("物料创建成功，ID：{}", material.getId());
        return material.getId();
    }
}
```

### 4.2 前端开发规范

#### 4.2.1 Vue 3 Composition API规范

**组件定义**：

```vue
<template>
  <el-card>
    <el-form :model="queryParams" inline>
      <el-form-item label="物料名称">

      <el-table-column prop="stockQty" label="库存数量" />
    </el-table>
  </el-card>

<script setup>
import { ref, reactive, onMounted } from 'vue'
import { getMaterialList } from '@/api/wms/material'

const queryParams = reactive({
  pageNum: 1,
  pageSize: 20,
  materialName: ''
})

const tableData = ref([])
const loading = ref(false)

  loading.value = true
  try {
    const { data } = await getMaterialList(queryParams)

  } finally {
    loading.value = false
  }
}

onMounted(() => {
  handleQuery()
})
</script>
```

## 5. 非功能性设计

### 5.1 性能设计目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| API响应时间（P95） | ≤500ms | 95%的请求响应时间不超过500ms |
| API响应时间（P99） | ≤1000ms | 99%的请求响应时间不超过1000ms |
| QPS | ≥1000 | 系统整体QPS不低于1000 |
| 并发用户数 | ≥500 | 支持500个并发用户同时在线 |
| 数据库连接池 | 20 | 每个服务最大20个连接 |

### 5.2 可用性设计

| 指标 | 目标值 | 实现方式 |
|------|--------|---------|
| 服务可用性 | ≥99.5% | 健康检查 + 自动重启 |
| 数据可靠性 | ≥99.99% | MySQL主从复制 + 每日备份 |
| RTO（恢复时间目标） | ≤30分钟 | Docker Compose快速重启 |
| RPO（恢复点目标） | ≤1小时 | 每小时增量备份 |

### 5.3 可扩展性设计

- **水平扩展**：通过Docker Compose scale命令扩展服务实例
- **垂直扩展**：调整JVM参数（-Xmx）增加内存
- **数据库扩展**：MySQL主从复制，读写分离
- **缓存扩展**：Redis Cluster集群模式

### 5.4 监控与运维

**监控指标**：
- 服务健康状态（/actuator/health）
- JVM内存使用率（/actuator/metrics/jvm.memory.used）
- 数据库连接池状态（/actuator/metrics/hikaricp.connections）
- API响应时间（/actuator/metrics/http.server.requests）
- 错误率（/actuator/metrics/http.server.requests?tag=status:5xx）

**日志管理**：
- 日志格式：JSON格式，包含traceId、userId、timestamp
- 日志级别：生产环境INFO，开发环境DEBUG
- 日志存储：Docker容器日志 + ELK（可选）
